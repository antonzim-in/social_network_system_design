# Система социальной сети для путешественников

## Описание проекта
Данный проект представляет собой системный дизайн социальной сети, ориентированной на путешественников. Платформа позволяет пользователям публиковать посты о путешествиях, ставить оценки, комментировать, подписываться друг на друга, искать популярные места и просматривать ленту активности. Проект разрабатывается в рамках курса по системному дизайну (System Design) и будет последовательно уточняться в следующих домашних заданиях.

## Функциональные требования
Система должна предоставлять следующий набор функций:

1. **Публикация постов**  
   - Пользователь может создать пост, содержащий:
     - фотографии (до 10 изображений);
     - текстовое описание (до 2000 символов);
     - привязку к географическому месту (координаты или название).
   - Пост можно редактировать и удалять.

2. **Взаимодействие с постами**  
   - **Реакции**: пользователь может ставить «лайк» (оценку) на пост, а также снимать его.
   - **Комментарии**: пользователь может оставлять комментарии к посту (до 500 символов), редактировать и удалять их.

3. **Социальные связи**  
   - Пользователь может подписываться на других путешественников и отписываться от них.
   - Подписки используются для формирования персональной ленты.

4. **Ленты**  
   - **Лента друзей** (на основе подписок) – посты в обратном хронологическом порядке.
   - **Общая лента** – посты всех пользователей (например, для поиска новых авторов).

5. **Поиск и навигация**  
   - Поиск популярных мест для путешествий (по названию или геопозиции).
   - Просмотр постов, привязанных к выбранному месту.

6. **Клиентские интерфейсы**  
   - Веб-версия (адаптивный дизайн под десктоп и мобильные браузеры).
   - Мобильные приложения (iOS, Android) с возможностью работы в офлайн-режиме (кэширование).

## Нефункциональные требования

### Аудитория приложения
- **Количество пользователей**  
  - DAU (дневная аудитория) через год: **10 000 000**.
  - Рост аудитории – линейный, после достижения 10 млн DAU продолжит расти.
- **Поведение пользователей**  
  - Среднее количество запросов на одного пользователя в день: **30**.
- **Регионы использования**  
  - Страны СНГ (Россия, Украина, Беларусь, Казахстан и др.).

### Особенности приложения
- **Сезонности**  
  - Пиковые нагрузки приходятся на летние месяцы (июнь–август) и новогодние праздники (декабрь–январь). Ожидается увеличение нагрузки до **30–50%** выше среднегодовой.
- **Хранение данных**  
  - Все основные данные (посты, комментарии, реакции, подписки, метаданные) хранятся постоянно.
  - Возможно агрегирование старых данных для аналитики, но без потери исходной информации.
- **Лимиты и ограничения**  
  - Максимальное количество подписчиков у пользователя: **1 000 000**.
  - Максимальный размер поста: текст – 2000 символов, фотографии – до 10 шт., каждая не более 10 МБ.
  - Максимальная длина комментария: 500 символов.
- **Временные ограничения**  
  - API для чтения (лента, поиск): **p95 ≤ 500 мс**.
  - API для записи (создание поста, комментария): **p95 ≤ 2 с**.
  - Доставка уведомлений: **не более 5 с**.

### Доступность и надёжность
- **Доступность** – **99,9%** (максимальное время простоя не более 8,76 часов в год).
- **Отказоустойчивость** – система должна сохранять работоспособность при выходе из строя отдельных узлов (горизонтальное масштабирование, репликация данных).
- **Безопасность**  
  - Защита от DDoS-атак.
  - Аутентификация и авторизация (OAuth 2.0 / JWT).
  - Шифрование канала связи (TLS 1.3).

### Масштабируемость
- Горизонтальное масштабирование всех компонентов (веб-серверы, сервисы, базы данных) для поддержки роста аудитории.

### Кроссплатформенность
- Поддержка последних версий браузеров: Chrome, Firefox, Safari, Edge.
- Мобильные приложения для iOS (iOS 15+) и Android (API level 21+).

## Оценка нагрузки

Для расчёта нагрузки используются следующие метрики:
- **DAU** = 10 000 000
- **Среднее количество запросов на пользователя в день** = 30
- **Количество секунд в сутках** = 86 400

### Общий RPS
```
RPS_total = DAU * avg_requests_per_day / 86400 = 10 000 000 * 30 / 86 400 ≈ 3472 RPS
```
### Распределение по подсистемам
Нагрузка разбита по трём ключевым подсистемам (посты, реакции, комментарии) и дополнительным операциям (подписки, профили, аутентификация и т.д.). Для каждой подсистемы приведены:
- **RPS** – количество запросов в секунду.
- **Средний размер запроса** – входящий трафик.
- **Средний размер ответа** – исходящий трафик.

#### Посты (CRUD, лента, поиск)
| Тип операции             | RPS   | Размер запроса | Размер ответа |
|--------------------------|-------|----------------|---------------|
| Получение ленты / поиск  | 1200  | 2 КБ           | 50 КБ         |
| Создание поста           | 30    | 5 МБ (фото)    | 1 КБ          |
| Чтение отдельного поста  | 200   | 0,5 КБ         | 40 КБ         |
| Редактирование / удаление| 50    | 1 КБ           | 0,5 КБ        |
| **Итого**                | **1480** | **средний ~10 КБ** | **средний ~45 КБ** |

*Примечание:* средние размеры учитывают мультимедиа и текстовые данные. Для GET-запросов преобладает исходящий трафик.

#### Реакции (лайки)
| Тип операции                  | RPS   | Размер запроса | Размер ответа |
|-------------------------------|-------|----------------|---------------|
| Постановка / снятие лайка     | 300   | 0,5 КБ         | 0,5 КБ        |
| Получение количества лайков   | 200   | 0,5 КБ         | 0,5 КБ        |
| **Итого**                     | **500** | **0,5 КБ**     | **0,5 КБ**    |

#### Комментарии
| Тип операции                   | RPS   | Размер запроса | Размер ответа |
|--------------------------------|-------|----------------|---------------|
| Создание комментария           | 80    | 1 КБ           | 0,5 КБ        |
| Получение комментариев к посту | 300   | 0,5 КБ         | 8 КБ          |
| Редактирование / удаление      | 20    | 1 КБ           | 0,5 КБ        |
| **Итого**                      | **400** | **средний ~0,7 КБ** | **средний ~5 КБ** |

#### Другие подсистемы (подписки, профили, аутентификация и пр.)
| Тип операций                   | RPS   | Размер запроса | Размер ответа |
|--------------------------------|-------|----------------|---------------|
| Подписки / отписки             | 150   | 0,5 КБ         | 0,5 КБ        |
| Профили (просмотр, редактирование) | 300 | 2 КБ      | 15 КБ         |
| Аутентификация / авторизация   | 200   | 1 КБ           | 2 КБ          |
| Загрузка / получение фото      | 442   | 200 КБ         | 300 КБ        |
| **Итого**                      | **1092** | **средний ~40 КБ** | **средний ~100 КБ** |

*Общий RPS по всем подсистемам:* 
```
1480 + 500 + 400 + 1092 = **3472 RPS** (совпадает с расчётным значением).
```
### Трафик (входящий и исходящий)
Для каждой подсистемы рассчитан средний трафик на основе RPS и средних размеров запросов/ответов.

| Подсистема   | Входящий трафик (МБ/с) | Исходящий трафик (МБ/с) |
|--------------|------------------------|-------------------------|
| Посты        | 14,8                   | 66,6                    |
| Реакции      | 0,25                   | 0,25                    |
| Комментарии  | 0,28                   | 2,0                     |
| Другие       | 43,7                   | 109,2                   |
| **Итого**    | **59,0**               | **178,0**               |

*Примечание:* трафик рассчитан как RPS × средний размер. Для подсистемы «Посты» учтены загрузки фото (30 запросов в секунду с размером 5 МБ), что даёт значительный вклад в входящий трафик.

### Одновременные соединения
Количество одновременно открытых соединений (при условии, что каждый активный пользователь поддерживает одно постоянное соединение) составляет примерно 10% от DAU:
```
Connections = DAU × 0,1 = 10 000 000 × 0,1 = 1 000 000
```

Полученные данные (RPS, трафик, количество соединений) будут использованы для выбора архитектурных решений, технологического стека и расчёта необходимых ресурсов в следующих домашних заданиях.
